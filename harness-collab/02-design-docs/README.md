# 技术设计文档目录（02-design-docs）

本目录存放所有功能的技术设计文档。设计文档是需求文档到代码实现之间的桥梁，也是 Kiro 生成代码的直接依据。

---

## 编写规范

### 何时创建设计文档

- 需求文档经开发者确认后，**必须**先创建设计文档，再开始编码
- Kiro 在需求分析阶段完成后，会自动引导开发者完成设计文档的创建
- 设计文档经开发者明确确认后，方可进入编码实现阶段（GATE-02 检查点）

### 文档结构要求

每份设计文档必须包含以下章节：

| 章节 | 是否必填 | 说明 |
|------|----------|------|
| 概述 | ✅ 必填 | 技术方案的一句话总结 |
| 架构设计 | ✅ 必填 | 组件关系图（Mermaid）和数据流向说明 |
| 接口定义 | ✅ 必填 | REST API 端点列表，包含请求/响应格式 |
| 数据模型 | ✅ 必填 | 实体类、DTO 和数据库表结构 |
| 技术选型 | ✅ 必填 | 框架、库、工具的选择说明 |
| 风险与注意事项 | ✅ 必填 | 潜在风险和技术细节 |

### 架构图要求

- 使用 Mermaid 语法绘制架构图，确保图表可在 Markdown 渲染器中显示
- 架构图必须体现四层架构（controller → service → domain ← repository）
- 组件关系图应清晰展示各层之间的依赖关系

### 接口定义要求

- 所有 REST API 端点必须在设计文档中定义
- 接口定义应包含：HTTP 方法、路径、请求参数、请求体、响应体、错误码
- 接口定义完成后，需同步到 `harness-collab/04-api-docs/` 目录

### 数据模型要求

- 实体类字段定义需包含：字段名、类型、约束、说明
- 数据库表结构需提供建表 SQL
- DTO 需说明与实体类的映射关系

### 文档命名规范

- 格式：`{功能英文名}-design.md`
- 使用小写字母和连字符（kebab-case）
- 示例：`user-management-design.md`、`order-payment-design.md`

## 与 Dev_Lifecycle 的关系

```
[需求分析阶段] → harness-collab/01-product-specs/
      ↓
  GATE-01 通过
      ↓
[技术设计阶段] ← 本目录文档在此阶段创建
      ↓
  GATE-02 检查点：设计文档是否存在？
      ↓ 通过
[编码实现阶段] → 基于设计文档生成代码
```

设计文档是 Kiro 生成代码的直接依据，代码中的类名、方法名、包路径都应与设计文档保持一致。

---

## 设计文档与代码的同步

当代码实现与设计文档出现偏差时（例如接口调整、数据模型变更），必须：

1. 更新设计文档中对应的章节
2. 更新 `harness-collab/04-api-docs/` 中的 API 文档
3. 更新 `harness-collab/func.md` 中的最后更新时间

---

## 模板使用

使用 `templates/design-doc-template.md` 作为起点创建新的设计文档：

```bash
# 复制模板到当前目录
cp templates/design-doc-template.md {功能英文名}-design.md
```

然后按照模板中的提示填写各章节内容。
