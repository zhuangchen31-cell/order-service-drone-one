# 需求文档目录（01-product-specs）

本目录存放所有功能的产品需求文档（PRD）。需求文档是 Dev_Lifecycle 的起点，也是 Kiro 开始生成代码的前置条件。

---

## 编写规范

### 何时创建需求文档

- 开始任何新功能开发前，**必须**先创建需求文档
- Kiro 在收到功能开发请求时，会自动引导开发者完成需求文档的创建
- 需求文档经开发者明确确认后，方可进入技术设计阶段（GATE-01 检查点）

### 文档结构要求

每份需求文档必须包含以下章节：

| 章节 | 是否必填 | 说明 |
|------|----------|------|
| 功能背景 | ✅ 必填 | 说明为什么需要这个功能，解决什么问题 |
| 目标 | ✅ 必填 | 明确的、可量化的功能目标 |
| 用户故事 | ✅ 必填 | 至少一个用户故事，包含验收标准 |
| 非功能性需求 | ✅ 必填 | 性能、安全、兼容性要求 |
| 排除范围 | ✅ 必填 | 明确不在本次迭代范围内的内容 |

### 验收标准格式

验收标准必须使用 Given-When-Then 格式：

```
- Given [前置条件]：描述系统或用户的初始状态
- When [触发动作]：描述用户执行的操作或系统触发的事件
- Then [预期结果]：描述系统应该产生的可观测结果
```

### 文档命名规范

- 格式：`{功能英文名}-spec.md`
- 使用小写字母和连字符（kebab-case）
- 示例：`user-management-spec.md`、`order-payment-spec.md`

## 使用场景

### 场景 1：新功能开发

1. 开发者向 Kiro 描述新功能需求
2. Kiro 引导开发者填写需求文档模板
3. 开发者确认需求文档内容
4. Kiro 基于需求文档开始技术设计阶段

### 场景 2：需求变更

1. 开发者描述需求变更内容
2. Kiro 更新对应的需求文档
3. 评估变更对设计文档和代码的影响
4. 同步更新 `func.md` 中的最后更新时间

### 场景 3：需求回溯

当需要了解某功能的原始需求时，直接查阅本目录下对应的需求文档，所有历史变更通过 Git 记录可追溯。

---

## 与 Dev_Lifecycle 的关系

```
[需求分析阶段] ← 本目录文档在此阶段创建
      ↓
  GATE-01 检查点：需求文档是否存在？
      ↓ 通过
[技术设计阶段] → harness-collab/02-design-docs/
```

需求文档是整个 Dev_Lifecycle 的基础，后续的设计文档、代码实现和测试验证都应与需求文档中的验收标准保持一致。

---

## 模板使用

使用 `templates/product-spec-template.md` 作为起点创建新的需求文档：

```bash
# 复制模板到当前目录
cp templates/product-spec-template.md {功能英文名}-spec.md
```

然后按照模板中的提示填写各章节内容。
